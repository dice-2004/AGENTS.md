## 1. 開発原則と基本ルール

### 1.0 指示の優先順位
- 本ファイルとユーザーの対話中の指示が矛盾する場合、原則として**直近の対話内での明示的指示**を優先します。
- ただし、Git操作の制限やセキュリティ、変更禁止対象など「(重要)」タグが付いたルールについては、ユーザーが明示的にそのルールの一時解除を宣言しない限り、本ファイルの規定を優先してください。
- **本ファイル自体の変更禁止**: 本ファイル（[本ファイル名を記述。例: AGENTS.md]）自体の内容は、ユーザーからの明示的な指示がない限りAI側で変更してはいけません。ルールの矛盾点や改善案に気づいた場合は、直接編集せずユーザーに提案してください。

### 1.1 言語とコミュニケーション
- **使用言語**: すべてのコミュニケーション（回答、プランの作成、コード内の解説等）は原則**日本語**で行います。
- **仕様優先の開発**: 新しいコンポーネントやアルゴリズムを実装する前に、必ず仕様書を作成して合意を得てから作業を開始します。
- **確認と質問**: 不明な仕様や設計判断が生じた場合は、独断で進めず、必ずユーザーに質問・確認を行います。
- **コード変更前の確認フロー (重要)**:
  - 新規の設計変更や機能追加など、開発方針に関わる変更を加える前に、必ずユーザーへ事前に確認を取り、明示的な承認・合意を得てから作業を開始してください。
  - その際、**具体的なコード差分やコードブロックの細部は提示せず**、代わりに「どのファイルのどの部分がどのような状態（あるいは問題）であったため、どのような方針で修正するのか」を、**コードを用いずに日本語で分かりやすく説明**してください。
  - ただし、実装したコードの動作検証中に見つかったバグ、不具合、不整合、環境起因のエラーなど、**バグが完全になくなるまでの修正作業については、ユーザーの許可を待たずに自律的に修正・解決し、その結果を事後に報告してください。**
  - **自律修正の試行回数制限**: 同一のエラーやバグに対する自律的な修正の試行が [試行回数の上限を記述。例: 3回] を超えても解決しない場合は、自律的な修正ループを中止してください。そこまでの試行内容とエラーログ、および現状の問題点をユーザーに報告し、次の指示を仰いでください。
  - **承認不要（自律実行可）の例**: [自律実行可能な軽微な作業を記述。例: 既存関数内のバグ修正、コメント追加、命名の統一、テストコードの追加]
  - **承認必須の例**: [承認が必要な設計変更を記述。例: 新規ファイル/モジュールの作成、外部パッケージの追加、既存の公開インターフェース(API/関数シグネチャ)の変更、インフラ・設定ファイルの変更]
  - **変更範囲の限定**: 依頼された作業の範囲を超えて、無関係なファイルやコードへ変更を加えないでください。作業中に範囲外の改善点に気づいた場合は、その場で直接変更せず、提案としてユーザーに報告してください。

### 1.2 コード品質
- **読みやすさとベストプラクティス**: 命名規則の遵守、適切な型ヒントの付与、および [使用言語を記述。例: Python] のベストプラクティス（[ベストプラクティス規格を記述。例: PEP 8]）に従います。
- **保守性**: 将来的な仕様変更やシステムの更新に対応しやすいよう、モジュール化と疎結合な設計を意識します。
- **リファクタリング**: コードの重複を避け、できるだけ綺麗で簡潔なコードにリファクタリングを行います。
- **解説の提供**: どのような実装を行ったのか、どのような意図で設計したのかを逐一解説します。
- **ドキュメントの整合性**: 既存のソースコードコメントや公式ドキュメントと矛盾しない実装を行います。
- **エラー時のテンプレート参照**: 実装したプログラムでエラーが発生した場合は、基準となるテンプレートやサンプル（[サンプルのファイルパスを記述。例: sample_submission/main.py]）の実装を最優先で確認し、エラーの出ない安全な実装を模倣・再現して解決を図ってください。
- **リンターチェックの考慮 (重要)**:
  - ローカル開発やCI環境での検証のため、コード作成時は常に型やフォーマットに細心の注意を払い、[プロジェクトで使用するツールを記述。例: Black/Flake8/Mypy] などの静的解析ツールが正常終了（Green）するように記述してください。
  - 必要に応じて、リンターの自動修正機能等を活用し、自動修正後は意図した動作が維持されているかを確認してください。
  - [一時的なスキップ方法を記述。例: --no-lint オプション] 等で静的解析を一時的にスキップする手段が提供されている場合でも、それはあくまで一時的な対処とし、最終的にはすべての静的解析・テストを通過するコードを提供する必要があります。

### 1.3 ドキュメント管理・作業ログ (重要)
- **初期起動時のドキュメント読み込み優先度**:
  - タスク開始直後（本ガイドラインファイルの読み込み後）は、まず [メインのREADMEファイルを記述。例: README.md] を読み、次にそれに紐づく各種仕様書・ドキュメント（[ドキュメントディレクトリを記述。例: docs/ 配下]）を読み込んでプロジェクトの全体像と設計仕様を理解した上で、最後に作業ログ（[作業ログディレクトリを記述。例: log/ 配下の作業ログ]）を確認し、最新の作業進捗や過去の修正履歴を把握するフローを厳守してください。
- **作業ログの記録**: 開発作業やコードの変更を行った際は、必ず作業ログファイルに作業内容、実装の意図、検証結果を記録してください。ログは [ログの管理方針を記述。例: 全体共通のログ（log/general.md）と、開発対象のモジュール別のログ（log/<モジュール名>.md）に分割して管理] してください。
- **仕様書との整合性**: 仕様書が存在する場合、コードの変更箇所と仕様書の間で矛盾がないようにしてください。コードを変更して仕様（入出力、クラス設計など）が変わった場合は、必ず関連する仕様書も同時に修正・更新してください。
- **作業中の再確認**: 特定のモジュール・機能に関する作業に着手する際は、そのセッション内で以前に該当ドキュメントを読んだかどうかに関わらず、対応する仕様書・詳細ドキュメントをその都度再確認してから着手してください。「以前読んだはず」という記憶だけを根拠に、詳細な仕様確認を省略しないでください（長時間のセッションでは、以前読んだ内容がコンテキストから失われている場合があります）。

### 1.4 定期的な品質レビュー (重要)
- **不確実性の明示**: 仕様やAPIの挙動、外部情報など確信が持てない事項については、断定的な発言を避け、その旨を明示した上でユーザーに確認を求めてください。推測を事実であるかのように断定することは避けてください。
- **セルフレビューの実施**: 数回の会話（目安: [レビュー頻度の会話回数を記述。例: 3〜5回] のやりとり）ごとに、これまでの作業内容を振り返り、品質レビューを実施してください。
- **レビューの方法**: サブエージェントの活用、別ファイルへの書き出し、コードの再読など、具体的なセルフレビューの手法は問いませんが、以下の観点を必ず確認してください。
  - 作成・変更したコードにバグや論理エラーがないか
  - 仕様書やドキュメントとの整合性が保たれているか
  - 発言した事実関係（日付、API仕様など）に誤りがないか
  - コード品質（命名、構造、エラーハンドリング）に問題がないか
- **長時間セッションでの指示維持**: 会話やタスクが長時間・多数回のツール呼び出しにわたる場合、[セルフチェックを行うタイミングの目安を記述。例: 10ターンごと、または大きなタスクフェーズの移行時] に、本ファイルの内容を再確認し、指示からの逸脱がないかセルフチェックしてください。
- **レビュー結果の報告**: 問題が見つかった場合は、修正を行った上でユーザーに報告してください。

### 1.5 完了報告のフォーマット
- 作業完了時や自律修正後の報告は、以下の形式に統一して報告してください。
  1. **変更内容の要約**: 何をどのように変更したか
  2. **変更理由・意図**: なぜその変更が必要だったか、設計の意図
  3. **検証方法と結果**: どのように動作検証を行い、どのような結果が得られたか
  4. **残課題・次のアクション**: 今後対応が必要な事項や、ユーザーへの提案（あれば）

### 1.6 報告・承認依頼のチャネル
- 通常の完了報告や作業ログは [報告先を記述。例: チャット上での応答] で行います。
- 緊急の承認依頼やエラー通知など、即時対応が必要なものは [専用チャネルを記述。例: 特定のチャットチャンネル、通知先] を用いて区別してください。

### 1.7 開発環境の選択および Git 操作の制限 (重要)
- **開発環境の選択と確認**:
  - 開発環境が [コンテナ環境や特定の仮想環境名を記述。例: VS Code Dev Containers] に構築されている場合、AI アシスタントは、コマンド実行やテスト検証を行う前に、必ずユーザーに対してその環境内であるかを確認し、合意のもとで指示または実行を行ってください。
- **Git 操作の制限**:
  - コミット（`git commit`）やプッシュ（`git push`）、プル（`git pull`）など、Git リポジトリに変化を与えるような変更操作は、環境内外を問わず AI 側では実行せず、必ずユーザーに作業を依頼してください。
  - 開発環境の制限（[例: コンテナ内からのSSH経由での操作失敗など]）により特定のGit操作が失敗する場合は、AI側で無理に対処しようとせず、[代替手段を記述。例: ホスト側のターミナルから操作を行うようユーザーに依頼する] など、適切な代替手段をユーザーに提案・依頼してください。

### 1.8 トラブルシューティングの遵守と追記
- **トラブルシューティング手順**: 開発作業や動作検証中に問題やエラーが発生した場合は、まず [トラブルシューティング用のドキュメントパスを記述。例: docs/troubleshooting.md] を確認し、該当するトラブル解決方法がある場合はそれに従って修正を行ってください。
- **解決策の記録**: トラブルシューティングに解決方法がない新しいトラブルが発生した場合は、自律的に解決を図った上で、そのトラブルの内容と解決方法をドキュメントの末尾に追記・記録してください。

### 1.9 セキュリティと機密情報の取り扱い (重要)
- APIキー、トークン、パスワード等の機密情報は、コード、コミットログ、作業ログ、チャット履歴のいずれにも平文で出力・記録してはいけません。
- 機密情報の受け渡しや利用は、[環境変数、シークレットマネージャー、特定の設定ファイル名などを記述。例: 環境変数や専用の設定ファイル] を通じてのみ行うように設計してください。

### 1.10 破壊的操作・本番影響操作の確認 (重要)
- ファイル/ディレクトリの削除、コンテナ・サービスの再起動や停止、データベースへの破壊的な変更（マイグレーション実行、レコード削除等）、外部の共有リソースへの影響を伴う操作を行う前には、必ずユーザーへ内容を説明し、明示的な承認を得てから実行してください。
- 誤操作時の影響範囲が大きい操作（[対象とする本番・共有環境を記述。例: 本番環境相当のコンテナ、共有DB]）は、可能な限りドライラン（実行せず影響のみ確認する）を先に提示してください。
